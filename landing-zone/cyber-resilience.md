# サイバーレジリエンス ランディングゾーン

> サイバー攻撃（ランサムウェア等）からの迅速な復旧と事業継続を実現する Azure 設計ガイダンス

## 前提条件

- 本リファレンスアーキテクチャは、サイバー攻撃を受けることを前提として、バックアップの論理的隔離・別環境への復旧・侵害システムのネットワーク隔離を実現するための設計を対象としています。
- 本アーキテクチャは各システム別ランディングゾーン（勘定系、為替決済系等）と組み合わせて使用する**横断的なセキュリティレイヤー**として位置づけられます。
- 金融庁「金融分野におけるサイバーセキュリティに関するガイドライン」（2024年10月公表）および FISC 安全対策基準 第13版のサイバーレジリエンス強化要件に準拠しています。
- オンプレミス環境との接続は ExpressRoute による閉域網接続を前提としています。

## システム概要

| 項目 | 内容 |
|------|------|
| システム名 | サイバーレジリエンス基盤 |
| 主な機能 | バックアップ隔離、サイバー攻撃からの復旧、自動ネットワーク隔離、フォレンジック支援 |
| FISC外部性 | **横断的基盤** — 全ワークロードの事業継続に影響 |
| 重要度 | **Tier 1（最高）** |
| 処理特性 | イベントドリブン（セキュリティアラート起点の自動対応） |
| 可用性要件 | 本番ワークロードの SLA に準ずる |

## ユースケース

- ランサムウェア感染時に、感染前のクリーンなバックアップデータから別環境へ迅速に復旧する
- サイバー攻撃検知時に、侵害されたシステムを自動的にネットワーク隔離し、影響範囲を最小化する
- バックアップデータを論理的に隔離し、攻撃者によるバックアップの暗号化・削除を防止する
- フォレンジック調査のための証拠保全と隔離環境での安全な解析を実施する
- 定期的なサイバー復旧訓練（Cyber Recovery Drill）により復旧手順の実効性を検証する

## FISC基準上の位置づけ

サイバーレジリエンスは FISC 第13版で強化された重点領域であり、金融庁ガイドラインの「2.3.3 データ保護」および「オペレーショナル・レジリエンス」要件に直接対応します。

**適用される主な基準**:
- 統20〜統24: 外部委託管理・クラウドリスク（責任分界の明確化）
- 統28: サプライチェーンセキュリティ（第13版新設）
- 実25〜実30: サイバーセキュリティ対策（検知・対応・復旧）
- 実39〜実45: バックアップ（論理的隔離・改ざん防止）
- 実71, 実73, 実73-1: DR・コンティンジェンシープラン
- 実150〜実153: AI安全対策（AI活用型攻撃への防御含む）

## アーキテクチャの特徴

### サブスクリプション分離によるアカウント設計

サイバーレジリエンスの実効性を確保するため、**4つのサブスクリプション**を分離して運用します。これにより、ワークロード環境が侵害された場合でもバックアップデータと復旧環境の安全性を保証します。

| サブスクリプション | 役割 | アクセス権限 |
|-------------------|------|-------------|
| **ワークロード** | 本番環境の各システム（勘定系等）が稼働 | 通常の運用チーム |
| **データバンカー** | 論理的に隔離されたバックアップ保管 | セキュリティ管理者のみ（MUA 必須） |
| **リストア** | 本番とは完全に別の復旧環境 | インシデント対応チーム |
| **フォレンジック** | 侵害されたシステムの証拠保全・解析 | セキュリティ調査チーム |

> **設計根拠**: ワークロードサブスクリプションの管理者権限が侵害された場合でも、データバンカーは別サブスクリプション（推奨: 別テナント）に配置されるため、バックアップデータへのアクセスは遮断されます。Azure Resource Guard による Multi-user Authorization (MUA) により、単一管理者によるバックアップ削除を防止します。

### バックアップの論理的隔離

ランサムウェアがバックアップにまで感染・暗号化するリスクに対応するため、以下の多層防御を実装します。

#### Immutable Vault（不変バックアップ）

```
┌─────────────────────────────────────────────────────────────────┐
│                    ワークロードサブスクリプション                    │
│  ┌──────────────┐    Azure Backup    ┌───────────────────────┐  │
│  │  本番 VM/DB   │ ──────────────→  │ Recovery Services Vault │  │
│  └──────────────┘    (日次/複数回)    │  (Soft Delete 有効)    │  │
│                                      └───────┬───────────────┘  │
└──────────────────────────────────────────────┼──────────────────┘
                                               │ Cross-subscription
                                               │ レプリケーション
┌──────────────────────────────────────────────┼──────────────────┐
│                    データバンカーサブスクリプション                  │
│                                              ▼                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │         Immutable Vault (不変ロック)                       │  │
│  │  ┌─────────────────────────────────────────────────────┐  │  │
│  │  │ ・不変性: 有効（Irreversible ロック推奨）              │  │  │
│  │  │ ・Soft Delete: Always-On（無効化不可）                │  │  │
│  │  │ ・保持期間: 最低 180日（規制要件に応じて延長）         │  │  │
│  │  │ ・MUA: Resource Guard による多人数認可必須            │  │  │
│  │  │ ・暗号化: CMK (Customer Managed Key)                 │  │  │
│  │  └─────────────────────────────────────────────────────┘  │  │
│  └───────────────────────────────────────────────────────────┘  │
│                                                                  │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │  Resource Guard (別テナント推奨)                            │  │
│  │  ・バックアップ管理者に Contributor 権限を付与しない        │  │
│  │  ・保護対象操作: 削除/保持期間短縮/Soft Delete 無効化      │  │
│  └───────────────────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────────────────────┘
```

#### Multi-user Authorization (MUA)

Azure Backup の重要操作（バックアップ削除、保持期間の短縮、Soft Delete の無効化）に対して、**Resource Guard** による多人数認可を強制します。

| 構成パターン | 隔離レベル | 推奨度 |
|-------------|----------|--------|
| 同一サブスクリプション | 最低 | 非推奨 |
| 別サブスクリプション / 同一テナント | 中 | 許容 |
| **別サブスクリプション / 別テナント** | **最高** | **推奨** |

> **実装ポイント**: Resource Guard は別テナントに配置し、Backup 管理者には Resource Guard に対する Contributor / Backup MUA Admin / Backup MUA Operator 権限を一切付与しません。重要操作の実行には Entra ID Privileged Identity Management (PIM) による Just-in-Time アクセス申請を必須とします。

### 侵害システムの自動ネットワーク隔離

サイバー攻撃検知時に、侵害されたシステムを自動的にネットワーク隔離します。

```
┌──────────────────────────────────────────────────────────────┐
│                    検知 → 隔離 フロー                         │
│                                                              │
│  ┌───────────┐    アラート    ┌──────────────────┐           │
│  │ Microsoft  │ ──────────→ │ Microsoft Sentinel │           │
│  │ Defender   │              │  (SIEM/SOAR)      │           │
│  │ for Cloud  │              └────────┬─────────┘           │
│  └───────────┘                        │                      │
│                                       │ Automation Rule      │
│  ┌───────────┐                        ▼                      │
│  │ Defender   │              ┌──────────────────┐           │
│  │ for        │ ──────────→ │  Logic Apps       │           │
│  │ Endpoint   │              │  (Playbook)       │           │
│  └───────────┘              └────────┬─────────┘           │
│                                       │                      │
│                          ┌────────────┼────────────┐        │
│                          ▼            ▼            ▼        │
│                   ┌──────────┐ ┌──────────┐ ┌──────────┐   │
│                   │ NSG 変更  │ │ Entra ID │ │ 管理者   │   │
│                   │ (Deny All)│ │ アカウント│ │ 通知     │   │
│                   │          │ │ 無効化   │ │ (Teams/  │   │
│                   │          │ │          │ │  Email)  │   │
│                   └──────────┘ └──────────┘ └──────────┘   │
└──────────────────────────────────────────────────────────────┘
```

#### 自動隔離アクション

| # | トリガー | アクション | 実装サービス |
|---|---------|----------|------------|
| 1 | Defender for Cloud: ランサムウェア検知 | 該当 VM/サブネットの NSG に Deny All ルールを追加 | Sentinel → Logic Apps |
| 2 | Defender for Endpoint: マルウェア検知 | エンドポイントのネットワーク隔離 | Defender for Endpoint API |
| 3 | Entra ID Protection: 高リスクサインイン | 侵害アカウントの無効化 + セッション失効 | Sentinel → Logic Apps |
| 4 | Azure Backup: 破壊的操作の検知 | バックアップの Soft Delete 保護有効化 + アラート | Defender for Cloud → Logic Apps |
| 5 | GuardRails: 異常な管理操作 | PIM アクセスの即時取消 + 承認フロー凍結 | Sentinel → Logic Apps |

> **承認ワークフロー**: 高影響アクション（サブネット全体の隔離等）には承認ワークフローを設定し、誤検知による業務影響を防止します。低影響アクション（単一 VM の隔離）は自動実行を推奨します。

### 別環境への復旧（リストア）

本番環境が依然として侵害されている可能性を考慮し、**本番とは完全に別の環境**での復旧を想定します。

#### 復旧手順の概要

| # | ステップ | 内容 | 自動化 |
|---|---------|------|--------|
| 1 | バックアップ健全性確認 | Defender for Cloud のランサムウェアスキャン結果によるリストアポイントの健全性評価 | 自動 |
| 2 | リストアサブスクリプション準備 | IaC（Bicep/Terraform）によるインフラの自動再構築 | 自動 |
| 3 | データリストア | データバンカーからの Cross-subscription リストア実行 | 半自動 |
| 4 | データ整合性検証 | リストアされたデータの改ざん検知・完全性検証 | 半自動 |
| 5 | アプリケーション接続切替 | 復旧 DB への接続設定変更・アプリケーション再起動 | **手動** |
| 6 | 業務検証 | 業務影響評価・リスク判断を経て本番切替 | **手動** |

> **設計根拠**: ステップ 5-6 を手動としているのは、復旧データの業務影響度評価やリスク判断が必要なためです。特に Tier 1 システムでは、自動切替による意図しないデータ不整合のリスクを避けるため、人間の判断を介在させます。

### フォレンジック調査

侵害されたシステムの証拠保全と安全な解析環境を提供します。

| 機能 | Azure サービス | 説明 |
|------|--------------|------|
| VM スナップショット取得 | Azure Managed Disk Snapshot | 侵害 VM の時点スナップショットを証拠として保全 |
| ネットワークキャプチャ | Azure Network Watcher | パケットキャプチャによるネットワークフォレンジック |
| ログの不変保管 | Azure Storage (Immutable Policy + Legal Hold) | 証拠ログの改ざん防止・法的保全 |
| メモリダンプ | Azure VM Extension | 揮発性証拠の収集 |
| 安全な解析環境 | フォレンジックサブスクリプション内の隔離 VNet | 侵害システムのコピーを隔離環境で安全に解析 |
| タイムライン分析 | Microsoft Sentinel | MITRE ATT&CK マッピングによる攻撃チェーンの可視化 |

> **チェーン・オブ・カストディ**: 証拠の完全性を維持するため、暗号化ハッシュとアクセスログを記録します。Azure Storage の Immutable Policy（WORM）と Legal Hold を併用して、法的手続きに耐える証拠保全を実現します。

### セキュリティ監視・検知

## 可用性・DR設計

### バックアップ戦略

| 対象 | バックアップ頻度 | 保持期間 | レプリケーション | 隔離方式 |
|------|---------------|---------|----------------|---------|
| Azure VM | 日次 + 4時間毎 | 180日以上 | GRS (Cross-Region) | Immutable Vault (データバンカー) |
| SQL MI | 自動（5-10分間隔） | PITR 35日 + LTR 1年以上 | Failover Group + GRS | Immutable Vault |
| Cosmos DB | 継続的バックアップ | PITR 30日 + 定期 LTR | マルチリージョン | Immutable Vault |
| Storage (Blob) | — | バージョニング + Immutable | GRS / RA-GRS | Immutable Policy + Legal Hold |
| Key Vault | 自動 | — | Soft Delete 90日 + Purge Protection | 別サブスクリプション |

### Defender for Cloud とバックアップの統合

Microsoft Defender for Cloud のランサムウェアアラートと Azure Backup を統合し、ランサムウェア検知時にバックアップのリカバリポイントを自動的に保護します。

| 機能 | 説明 |
|------|------|
| リストアポイント健全性評価 | Defender for Servers のシグナルによりバックアップスナップショットのランサムウェア感染を検知 |
| 自動保護強化 | ランサムウェアアラート発生時に Logic Apps でリカバリポイントの有効期限を自動延長 |
| クリーンリストアポイント特定 | 感染前の安全なリストアポイントを迅速に特定し、復旧時間を短縮 |

### 復旧目標

| シナリオ | RTO | RPO | 復旧方式 |
|---------|-----|-----|---------|
| 単一 VM のランサムウェア感染 | 4時間以内 | 4時間（バックアップ頻度に依存） | Immutable Vault からリストア |
| サブネット全体の侵害 | 8-24時間 | 4時間 | リストアサブスクリプションへの全体復旧 |
| リージョン障害 + サイバー攻撃 | 24-48時間 | 4時間 | Cross-Region Restore（ペアリージョン） |
| Tier 1 システム（勘定系等） | 各 LZ の RTO に準拠 | 各 LZ の RPO に準拠 | 各 LZ の DR 設計と連携 |

## ネットワーク設計

### サブスクリプション別ネットワーク分離

サイバーレジリエンス関連のサブスクリプションは、ワークロードの Hub-Spoke ネットワークとは**独立した VNet** を使用し、必要最小限の接続のみを許可します。

| サブスクリプション | VNet CIDR | Hub 接続 | インターネット |
|-------------------|-----------|---------|-------------|
| データバンカー | 10.32.0.0/16 | **なし**（完全隔離） | **なし** |
| リストア | 10.33.0.0/16 | 必要時のみ Peering | 制限付き（パッチ適用のみ） |
| フォレンジック | 10.34.0.0/16 | **なし**（完全隔離） | **なし** |

> **設計根拠**: データバンカーは Azure Backup の管理プレーン経由でのみアクセスし、データプレーンのネットワーク接続は不要です。フォレンジック環境は完全に隔離し、証拠の汚染を防止します。

### NSG 自動隔離ルール

Sentinel Playbook により適用される隔離ルールの例:

| 優先度 | 方向 | アクション | ソース | 宛先 | ポート | 目的 |
|-------|------|----------|--------|------|--------|------|
| 100 | Inbound | Deny | Any | 侵害サブネット | Any | 外部からのアクセス遮断 |
| 101 | Outbound | Deny | 侵害サブネット | Any | Any | ラテラルムーブメント防止 |
| 102 | Inbound | Allow | フォレンジック VNet | 侵害サブネット | 22, 3389 | 調査アクセスのみ許可 |

## 監視・オブザーバビリティ

### Sentinel 検知ルール

| ルール名 | データソース | 検知ロジック | 重大度 |
|---------|------------|------------|--------|
| ランサムウェア指標検知 | Defender for Cloud | ランサムウェアシグネチャ・行動パターン一致 | Critical |

| 異常な大量ファイル暗号化 | Defender for Endpoint | 短時間の大量ファイル変更・暗号化パターン | High |
| 特権アカウント異常使用 | Entra ID | 通常時間外 / 異常 IP からの特権操作 | High |
| セキュリティツール無効化 | Azure Activity Log | Defender / Sentinel / NSG の無効化・変更 | Critical |
| Event Log クリア | Windows Security Log | セキュリティイベントログの消去 | High |

## サイバー復旧訓練（Cyber Recovery Drill）

定期的な訓練により、復旧手順の実効性と所要時間を検証します。

### 訓練シナリオ

| # | シナリオ | 頻度 | 検証項目 |
|---|---------|------|---------|
| 1 | 単一 VM ランサムウェア感染 → 隔離 → リストア | 四半期 | 自動隔離の動作確認、リストア所要時間 |
| 2 | バックアップ削除試行 → MUA ブロック確認 | 半年 | Resource Guard の有効性、PIM フロー |
| 3 | Tier 1 システム全体復旧（リストアサブスクリプション） | 年次 | IaC 再構築 + データリストア + 業務検証の全手順 |
| 4 | Cross-Region Restore（ペアリージョン復旧） | 年次 | GRS レプリカからの復旧、RPO/RTO 達成確認 |

> **訓練環境**: Azure Chaos Studio を活用してランサムウェア感染をシミュレートするフォールトインジェクションを実施し、実際の復旧手順を検証します。訓練結果は FISC 監査証跡として記録します。

## Azure Policy によるガバナンス

サイバーレジリエンス要件を Azure Policy で強制適用します。

| ポリシー | 効果 | 対象 |
|---------|------|------|
| Recovery Services Vault に Immutable Vault を強制 | DeployIfNotExists | 全サブスクリプション |
| Soft Delete の無効化を拒否 | Deny | Recovery Services Vault |
| GRS / ZRS を強制（LRS 拒否） | Deny | Recovery Services Vault |
| MUA (Resource Guard) 未設定の Vault を検出 | Audit | Recovery Services Vault |
| Defender for Cloud の有効化を強制 | DeployIfNotExists | 全サブスクリプション |
| Defender for Servers Plan 2 を強制 | DeployIfNotExists | VM サブスクリプション |
| NSG フローログの有効化を強制 | DeployIfNotExists | 全 NSG |
