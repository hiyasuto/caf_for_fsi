# インターネットバンキング ランディングゾーン

> 個人・法人向けインターネットバンキングのAzure設計ガイダンス

## システム概要

| 項目 | 内容 |
|------|------|
| システム名 | インターネットバンキング（IB） |
| 主な機能 | 残高照会、振込・送金、振替、各種届出、ローン申込 |
| FISC外部性 | 各金融機関の判断による（主要チャネルの場合は外部性大） |
| 重要度 | **Tier 2〜3** |
| 処理特性 | Web/API（HTTPS）、24時間365日運用 |
| 可用性要件 | 99.95%以上 |

## FISC基準上の位置づけ

インターネットバンキングはFISC基準の「ダイレクトチャネル」に分類されます。実112〜実117のインターネット・モバイルサービス固有の基準が適用されます。

## Azureサービス構成

| コンポーネント | Azureサービス | 構成 | 理由 |
|-------------|-------------|------|------|
| Webアプリ | App Service (Premium v3) / AKS | 可用性ゾーン + Autoscale | 24/365運用・負荷変動対応 |
| CDN | Azure Front Door | グローバルPOP | 静的コンテンツ配信・レイテンシ最適化 |
| WAF | Azure Front Door WAF | OWASP Core Rule Set | Webアプリ攻撃防御 |
| Bot対策 | Azure Front Door Bot Protection | ボット検知 | 不正ログイン試行防御 |
| 認証基盤 | Microsoft Entra External ID | B2C | 個人・法人顧客の認証 |
| MFA | Microsoft Entra MFA | SMS/アプリ/FIDO2 | 実112：不正使用防止 |
| セッション管理 | Azure Cache for Redis | 可用性ゾーン | セッション・OTP管理 |
| DB | Azure SQL Database (Business Critical) | 可用性ゾーン | 取引履歴・設定 |

## FISC実務基準対応（インターネット・モバイルサービス）

| FISC基準 | 要件 | Azure実装 |
|---------|------|----------|
| 実112 | 不正使用の防止 | Entra MFA + Conditional Access + Bot Protection |
| 実113 | 利用状況の確認手段 | ログイン履歴・取引履歴の表示機能 |
| 実115 | 顧客対応方法の明確化 | Azure Bot Service（FAQ）+ コールセンター連携 |
| 実117 | オンライン口座開設の本人確認 | Entra External ID + eKYC連携 |
| 実1 | パスワード等の保護 | パスワードレス認証（FIDO2）推奨 |
| 実8 | 本人確認 | リスクベース認証（Entra ID Protection） |
| 実9 | ID不正使用防止 | アカウントロックアウト + 異常検知 |

## セキュリティ設計（多層防御）

```
Layer 1: Azure Front Door + DDoS Protection (L3/L4防御)
    ↓
Layer 2: Azure WAF (L7防御 — SQLi, XSS, CSRF)
    ↓
Layer 3: Bot Protection (不正ログイン試行防御)
    ↓
Layer 4: 認証・認可 (Entra External ID + MFA)
    ↓
Layer 5: アプリケーション層 (入力検証・セッション管理)
    ↓
Layer 6: ネットワーク分離 (NSG + Private Link)
    ↓
Layer 7: データ保護 (TDE + CMK + TLS 1.2+)
```

## 可用性・DR設計

| 要件 | 設計 |
|------|------|
| **RTO** | < 15分 |
| **RPO** | < 5分 |
| 構成 | 可用性ゾーン3ゾーン + Azure Front Door |
| リージョンDR | 西日本 Cold Standby（IaCで迅速再構築） |
| スケーリング | App Service Autoscale（CPU/リクエスト数ベース） |
