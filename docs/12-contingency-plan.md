# 12 — コンティンジェンシープラン

> FISC実務基準（実73）＋ コンティンジェンシープラン策定のための手引書（第5版）→ Azure Reliability / Operational Excellence

## 概要

本ドキュメントは、パブリッククラウドサービス利用に伴うコンティンジェンシープランの策定・維持管理に関するガイダンスです。FISC安全対策基準（第13版）の実73（コンティンジェンシープランの策定）およびFISCコンティンジェンシープラン策定のための手引書（第5版）の要件を、Azure のサービス・機能にマッピングして実現します。

> **関連ドキュメント**: バックアップ・DR アーキテクチャの設計は [04. 信頼性・事業継続](04-reliability.md) を、障害シナリオの策定と演習は [10. 障害シナリオと演習](10-disaster-exercise.md) を、インシデント発生時の対応手順は [11. インシデント対応計画](11-incident-response.md) を参照してください。

## 2. RTO/RPO 設計ガイダンス

### 2.1 FISC 分類に基づく RTO/RPO 目標

| 重要度 | FISC分類 | RTO 目標 | RPO 目標 | Azure 構成パターン |
|-------|---------|---------|---------|-------------------|
| **最高** | 重大な外部性を有するシステム | < 1時間 | ほぼゼロ | 可用性ゾーン + リージョンペア + 同期レプリケーション |
| **高** | 基幹業務系 | < 4時間 | < 1時間 | 可用性ゾーン + ASR（リージョン間） + 非同期レプリケーション |
| **中** | 基幹業務系以外（重要） | < 24時間 | < 4時間 | 可用性セット + Azure Backup（GRS） |
| **標準** | 一般業務系 | < 72時間 | < 24時間 | Azure Backup（LRS/GRS） |

### 2.2 金融システム別 RTO/RPO マッピング

| 対象システム | 重要度 | RTO | RPO | Azure DR 構成 | 関連 LZ |
|------------|-------|-----|-----|-------------|---------|
| 勘定系（コアバンキング） | 最高 | < 1時間 | ほぼゼロ | AZ + リージョンペア + SQL Always On | [勘定系](../landing-zone/core-banking.md) |
| 為替・決済系 | 最高 | < 1時間 | ほぼゼロ | AZ + Failover Group + Front Door | [決済系](../landing-zone/payment-settlement.md) |
| インターネットバンキング | 高 | < 2時間 | < 15分 | AZ + ASR + SQL Geo-replication | [IB](../landing-zone/internet-banking.md) |
| 対外接続系 | 最高 | < 1時間 | ほぼゼロ | 冗長 ExpressRoute + AZ | [対外接続](../landing-zone/external-connectivity.md) |
| 情報系・DWH | 中 | < 24時間 | < 4時間 | Azure Backup GRS | [DWH](../landing-zone/dwh-bi.md) |
| 開発基盤 | 標準 | < 72時間 | < 24時間 | Azure Backup LRS | [開発基盤](../landing-zone/engineering-platform.md) |

### 2.3 RTO/RPO の検証方法

| 検証項目 | 方法 | Azure ツール | 頻度 |
|---------|------|-------------|------|
| フェイルオーバー時間 | ASR テストフェイルオーバー | Azure Site Recovery | 半期 |
| DB 切替時間 | Failover Group 強制フェイルオーバー | Azure SQL | 半期 |
| バックアップ復元時間 | リストアテスト | Azure Backup | 四半期 |
| エンドツーエンド復旧時間 | 総合 DR 演習 | → [10. 演習](10-disaster-exercise.md) L2 | 半期 |

## 3. リスク別対応方針

### 3.1 自然災害

| リスクシナリオ | 対応方針 | Azure 構成 |
|-------------|---------|-----------|
| **大規模地震（南海トラフ等）** | 東日本・西日本リージョンの DR ペア構成 | ASR リージョン間レプリケーション + Front Door |
| **台風・水害** | 可用性ゾーンによる同一リージョン内冗長化 | 3 AZ 構成 + Zone Redundant サービス |
| **広域災害（東日本・西日本同時）** | 海外リージョンへの縮退運転 | 東南アジアリージョンの Cold Standby |
| **通信インフラ障害** | 複数経路の通信手段確保 | ExpressRoute 2回線 + VPN バックアップ |

### 3.2 大規模システム障害

| リスクシナリオ | 対応方針 | Azure 構成 |
|-------------|---------|-----------|
| **単一ゾーン障害** | 自動フェイルオーバー | Zone Redundant 構成 |
| **リージョン全体障害** | DR リージョンへの切替 | ASR + SQL Failover Group |
| **Azure 全体障害** | オンプレミスへの縮退 | Azure Arc + ハイブリッド構成 |
| **データ破損** | ポイントインタイムリストア | Azure SQL PITR / Backup |
| **構成変更起因** | IaC によるロールバック | Bicep/Terraform + Azure Pipelines |

### 3.3 サイバー攻撃

| リスクシナリオ | 対応方針 | Azure 構成 |
|-------------|---------|-----------|
| **ランサムウェア** | 不変バックアップからのリストア | Immutable Vault + クリーン環境再構築 |
| **大規模 DDoS** | 自動緩和 + CDN 活用 | DDoS Protection Standard + Front Door |
| **サプライチェーン攻撃** | 隔離環境での調査・復旧 | 隔離サブスクリプション + フォレンジック環境 |
| **内部犯行** | アクセスログに基づく影響範囲特定 | Entra ID ログ + Sentinel 調査 |

> **詳細**: インシデント発生時の具体的な対応手順は [11. インシデント対応計画](11-incident-response.md) を参照してください。

## 5. 縮退運転の設計

### 5.1 縮退運転モードの定義

| モード | 稼働レベル | 提供機能 | 停止機能 | 切替基準 |
|-------|-----------|---------|---------|---------|
| **通常運転** | 100% | 全機能 | なし | — |
| **縮退レベル1** | 80% | 重要業務のみ | 参照系・レポーティング | 単一サービス障害 |
| **縮退レベル2** | 50% | 最重要業務のみ | 非重要業務全般 | 複数サービス障害 |
| **縮退レベル3** | 最小限 | 勘定照会・ATM のみ | オンライン取引全般 | リージョン障害 |
| **全面停止** | 0% | なし | 全機能 | 壊滅的障害 |

### 5.2 縮退運転の Azure 実装

| 縮退操作 | Azure 実装 | 自動/手動 |
|---------|-----------|----------|
| 非重要サービスの停止 | Azure Front Door ルーティング変更 | 手動（承認付き） |
| 読み取り専用モード | Azure SQL Read-Only Replica への切替 | 自動（フェイルオーバー時） |
| キャパシティ削減 | VMSS / AKS のスケールダウン | 手動 |
| 外部接続の制限 | API Management スロットリング強化 | 自動（ポリシーベース） |
| 静的ページへの切替 | Azure Static Web Apps / CDN フォールバック | 自動（Front Door ヘルスプローブ） |

### 5.3 優先復旧順序

```
復旧優先順序（金融機関の一般的なパターン）

第1優先（< 1時間以内に復旧）
├── 勘定系（コアバンキング）
├── 為替・決済系
└── 対外接続系（全銀/SWIFT）

第2優先（< 4時間以内に復旧）
├── インターネットバンキング
├── ATM 系
└── モバイルバンキング

第3優先（< 24時間以内に復旧）
├── 融資系
├── リスク管理系
└── AML/KYC

第4優先（< 72時間以内に復旧）
├── 情報系・DWH/BI
├── 市場系（非リアルタイム部分）
└── 開発基盤
```

### 7.1 報告基準

| 報告種別 | 基準 | 報告先 | 期限 |
|---------|------|-------|------|
| **障害報告（速報）** | 重要業務に影響する障害発生時 | 金融庁 | 速やかに |
| **障害報告（詳報）** | 重要業務に影響する障害の詳細 | 金融庁 | 所定期限内 |
| **サイバーインシデント報告** | サイバー攻撃による影響発生時 | 金融庁 / 金融ISAC | 速やかに |
| **定期報告** | CP の整備・更新状況 | 金融庁（検査対応） | 定期 |

### 7.2 報告に必要な情報と Azure データソース

| 報告項目 | 内容 | Azure データソース |
|---------|------|-------------------|
| 障害概要 | 発生日時・検知日時・影響範囲 | Azure Service Health / Monitor アラート |
| 影響業務 | 影響を受けた業務・顧客数 | Application Insights / Resource Health |
| 原因 | 障害の原因（暫定・確定） | Azure Service Health PIR / 変更分析 |
| 対応状況 | 実施した対応・現在の状況 | Sentinel インシデントタイムライン |
| 復旧見込み | 復旧の見通し・RTO 達成状況 | Azure Monitor メトリクス |
| 再発防止策 | 根本原因に対する改善計画 | Azure DevOps Work Items |
