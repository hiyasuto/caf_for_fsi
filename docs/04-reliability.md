# 04 — 信頼性・事業継続

> FISC実務基準（実39〜実45, 実71〜実74, 実102〜実105）→ Azure Reliability（設計・アーキテクチャ）

## 概要

FISC基準におけるバックアップ設計・DRアーキテクチャ要件を、Azure WAFの**信頼性の柱**（Reliability Pillar）に基づいて実現します。本ドキュメントは「**どう設計するか**」に焦点を当てており、実73・実73-1の詳細な運用プロセスは専用ドキュメントに委譲しています。

## 1. バックアップ（実39〜実45）

### 実39: データファイルのバックアップ

**FISC要件**: データファイルのバックアップを確保すること。

| 対策 | Azureサービス | 説明 |
|------|-------------|------|
| 統合バックアップ | Azure Backup | VM、SQL、ファイル、BLOBの統合バックアップ |

### 実74: バックアップサイトの保有

**FISC要件**: データセンター等が災害等により機能しなくなった場合に備えるため、業務の優先度を考慮したバックアップサイトを保有すること。

FISC第13版では、バックアップサイトの運営形態として自営・共同利用・相互利用・代行処理の4類型を定義しています。クラウド利用の場合は、ゾーン・リージョンの選定によりこれらを実現します。

| 対策区分 | Azureサービス | 説明 |
|---------|-------------|------|
| リージョン冗長 | Azure Paired Regions | 地理的に離れたリージョンペアによるリスク分散（300km以上の距離） |
| ゾーン冗長 | Availability Zones | 同一リージョン内の独立したデータセンター群（電源・冷却・ネットワーク独立） |
| サイト切替 | Azure Site Recovery（ASR） | 本番サイトからバックアップサイトへのフェールオーバー自動化 |
| グローバル負荷分散 | Azure Front Door / Traffic Manager | 複数リージョンへのトラフィック分散・自動フェールオーバー |
| データレプリケーション | Azure SQL geo-replication / Cosmos DB | データベースのリージョン間リアルタイム複製 |
| ストレージ冗長 | GRS / GZRS | 地理冗長ストレージによるデータのリージョン間複製 |

#### クラウド固有の確認事項（FISC参考2・参考3準拠）

| 確認事項 | 確認内容 | Azure対応 |
|---------|---------|----------|
| サービス継続仕様 | RPO/RTO、フェールオーバー判定基準・切替手段 | ASR復旧計画、Azure Front Doorヘルスプローブ |
| フェールオーバー制御 | 利用者による判断の可否、接続変更対応 | ASR手動/自動フェールオーバー、DNS切替 |
| フェールバック手順 | クラウド事業者の判定基準、切替手段 | ASR再保護・フェールバック機能 |
| ライセンス有効化 | バックアップサイト切替後のライセンス手順 | Azure Hybrid Benefit、予約インスタンスの移動 |
| リソース競合対策 | 災害時の一斉立ち上げによるリソース逼迫 | Capacity Reservations（容量予約）による確保 |

> **リソース競合への対策（参考3）**: 大規模災害時には複数のクラウド利用者が一斉にバックアップサイトの仮想マシンを立ち上げるため、リソースが逼迫する懸念があります。Azure **Capacity Reservations**（オンデマンド容量予約）や**ホットスタンバイ構成**（常時稼働）、**Active-Active構成**（異なるリージョン間での業務分散稼働）により、リソース確保を保証できます。

## 3. 障害の早期発見・回復機能（実102〜実105）

FISC第13版「システムの信頼性向上対策 (4) 障害の早期発見・回復機能」に対応し、障害の監視・検出・切分け・縮退運転・取引制限をAzureサービスで実現します。

### 実102: システム運用状況の監視機能

**FISC要件**: 障害の早期発見・回復のため、コンピュータシステムの運用状況（稼働状態、停止状態、エラー状態）を監視する機能を設けること。

| 監視方式 | Azureサービス | 説明 |
|---------|-------------|------|
| インフラ監視 | Azure Monitor（メトリクス） | CPU・メモリ・ディスクI/O・ネットワーク等の稼働状況監視 |
| アプリケーション監視 | Application Insights | トランザクション監視・応答時間・エラー率のリアルタイム把握 |
| ネットワーク監視 | Azure Network Watcher | 回線・NSG・VPN Gateway・ExpressRoute等の運用状況集中監視 |
| ログ集中管理 | Log Analytics ワークスペース | 複数リソースのログ一元管理・KQLによる横断分析 |
| 統合ダッシュボード | Azure Monitor Workbooks / Grafana | 運用状況の集中監視ダッシュボード |
| アラート | Azure Monitor Alerts | 閾値ベース・動的閾値による異常検知と自動通知 |
| リモート監視 | Azure Arc | オンプレミス・マルチクラウド環境を含む統合リモート監視 |

> **マネージドサービスの監視（参考2・3）**: PaaS/サーバーレス等のマネージドサービスでは、個々のコンポーネント監視よりも**サービス監視**（トランザクション発生状況等によるサービス正常性の確認）の重要性が高まります。Application Insightsの**可用性テスト**やAzure Front Doorの**ヘルスプローブ**を活用し、エンドツーエンドのサービス監視を実現します。オートスケーリングや自動フェールオーバー機能を備えたサービスに対しても、それに対応した監視を整備します。

### 実103: 障害の検出及び障害箇所の切分け機能

**FISC要件**: コンピュータシステムに発生する各種障害を的確に検出し、障害箇所を切り分ける機能を設けること。

| 機能区分 | Azureサービス | 説明 |
|---------|-------------|------|
| 異常検出 | Azure Monitor Alerts（動的閾値） | 機械学習ベースの異常検出・ベースラインからの逸脱検知 |
| ロギング | Azure Monitor Logs / 診断設定 | エラー発生時の詳細情報・稼働トレースのロギング |
| 分散トレーシング | Application Insights（分散トレース） | マイクロサービス間のリクエストフロー追跡・障害箇所特定 |
| 依存関係マッピング | Application Insights（アプリケーションマップ） | コンポーネント間の依存関係可視化・障害影響範囲の把握 |
| 障害診断 | Azure Resource Health | 各リソースの正常性ステータス・プラットフォーム起因障害の識別 |
| サービス正常性 | Azure Service Health | Azure基盤側の障害情報・計画メンテナンス・正常性アドバイザリー |
| ネットワーク診断 | Network Watcher（接続トラブルシューティング） | ネットワーク経路の障害箇所切分け・パケットキャプチャ |

> **クラウド固有の確認事項**: クラウド環境では仮想OS基盤等、利用者が直接監視できないレイヤーがあります。Azure **Resource Health**でプラットフォーム起因の障害を識別し、**Service Health**でAzure基盤側の障害情報を取得することで、自社アプリケーション起因か基盤起因かの切分けを迅速に行えます。

### 実103-1: 冗長構成・バックアップ構成の正常機能確認（第13版新設）

**FISC要件**: 冗長構成やバックアップ構成を採用しているシステムについては、その構成が正しく機能するための措置を講じ、信頼性を高めること。

| 対策区分 | Azureサービス | 説明 |
|---------|-------------|------|
| 障害注入テスト | Azure Chaos Studio | 疑似障害を注入し、冗長構成の自動切替を検証 |
| フェールオーバー訓練 | Azure Site Recovery（テストフェールオーバー） | 本番影響なしでのDR切替検証 |
| 正常性検証 | Azure Load Balancer ヘルスプローブ | 待機系を含む冗長構成の常時正常性確認 |
| 手動切替手順 | Azure Automation Runbook | 自動切替失敗時の手動フェールオーバー手順の整備 |
| 構成変更時検証 | Azure DevOps パイプライン | インフラ変更時の自動テストによる冗長構成の検証 |

> **サイバー攻撃対策**: FISC第13版では、サイバー攻撃により本番システムとバックアップシステムが同時に使用不能になる可能性の低減対策も求めています。バックアップを**イミュータブルストレージ**に保管し、本番環境とは異なる認証情報でアクセス制御することで、同時侵害リスクを低減します。

### 実104: 障害時の縮退・再構成機能

**FISC要件**: 障害時に一部の処理を中断しても、システム全体を停止させることなく運転を続行させるため、機能を縮小しシステムを再構成する機能を設けること。

| 対策区分 | Azureサービス | 説明 |
|---------|-------------|------|
| 取引閉塞 | Azure API Management（レート制限・サーキットブレーカー） | 障害部分に関する取引の閉塞・トラフィック制御 |
| 障害部分切離し | Azure Load Balancer / Traffic Manager | 障害ノードのヘルスプローブ失敗による自動切離し |
| 自動再構成 | Azure VM Scale Sets / AKS | 障害ノード検知後の自動スケールアウト・Pod再スケジューリング |
| 縮退運転 | Azure Feature Flags（App Configuration） | 機能フラグによる段階的な機能縮退・非重要機能の停止 |
| 障害復旧後の再組込み | Azure Load Balancer | 復旧ノードのヘルスプローブ成功による自動再組込み |

> **オペレータ介入の最小化**: FISC基準では縮退・再構成時のオペレーションのパターン化とオペレータ介入の最小化を推奨しています。AKSの**Self-healing**（Pod自動再起動）やVM Scale Setsの**自動修復ポリシー**により、障害検知→切離し→再構成を自動化できます。

### 実105: 障害時の取引制限機能

**FISC要件**: 障害の影響を極小化するため、障害の状況に応じ一部取引を稼働中のシステムから切り離すなどの取引制限機能を設けること。

| 制限単位 | Azureサービス | 実装方法 |
|---------|-------------|---------|
| 業務単位 | Azure API Management | APIポリシーによる業務別のアクセス制御・レート制限 |
| サービス単位 | Azure Front Door（ルーティングルール） | サービス別のトラフィック制御・メンテナンスページ表示 |
| リージョン単位 | Azure Traffic Manager | 地域別のトラフィック制御・障害リージョンの切離し |
| エンドポイント単位 | Azure Application Gateway | パスベースルーティングによる特定機能の制限 |
| 機能単位 | Azure App Configuration（Feature Flags） | 機能フラグによるリアルタイムな機能ON/OFF制御 |
